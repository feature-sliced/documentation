### Purpose over essence

In the diagrams above, every element is annotated with its purpose â€” whether it represents logic, data or UI. It makes for a convenient grouping of code by its technical purpose. In FSD, we maintain this grouping through segments. Segments are just folders inside of slices. Some common segment names are `ui/`, `model/`, and `api/`. 

- `ui` contains everything that involves presentation â€” UI components, data formatting functions, hooks for UI logic, etc.
- `model` contains the data and business logic â€” stores for certain types of objects, data mappers, hooks for business logic, etc.
- `api` contains everything to do with communicating with external APIs â€” API calls, queries, mutations.

This kind of grouping is beneficial because it allows you to predict where a certain module will be in the file system. If youâ€™re looking for API query hooks, you donâ€™t have to dig through the `hooks/` folder, ignoring all the UI and logic hooks.

Youâ€™re not limited to these three segments, you could define your own, just make sure that they reflect the purpose of the code (the why), not the essence (the what).

### Business-less layers

Talking about segments is a good segue into the two layers-exceptions: App and Shared. ### Talking to the backend

Now that most of our application is laid out, we need to fill it with some data. Our backend provides, but how do we retrieve and store it?

We already touched on the API and Model segments in slices. They exist for this exact purpose, and they make the most sense on the Entities layer, since thatâ€™s where you need to fetch *and* store data. The API segment is also commonly found in Features, usually containing mutations, that is, operations that change the data on the backend.

With some tech stacks you might not even need the Model segment for storage. For example, [TanStack Query](https://tanstack.com/query/latest) stores the server-side data in cache, allowing you to request a fetch and retrieve the data at the same time. Other stacks may have these operations separate, such as [Redux](https://redux.js.org/), where you dispatch an action to fetch and then use a selector to extract the fetched data from the store. In this case, the fetching code goes in API and the selector/reducer go to Model.
